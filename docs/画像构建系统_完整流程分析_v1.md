# 画像构建系统 · 完整流程分析

> 版本：v1.0 | 作者：AI分析 | 日期：2026-03-17
>
> 本文档基于项目代码（`frontend/src/modules/profile-helper/` + `backend/`）的完整阅读，梳理用户从进入画像系统到画像建立完成的全部路径、分支、小闭环与大闭环。

---

## 总体架构一句话

> **一个 AI Agent（LLM）与用户通过 Block 协议对话，按需加载 Skill 文件，逐段采集数据，每次采集完成后写入画像 Markdown 文件，最终构成一份结构化科研人员画像。**

整个系统由三层构成：

| 层 | 技术实现 | 职责 |
|---|---|---|
| 前端 UI | React + ChatWindow + BlockRenderer | 渲染对话、呈现交互控件（选项/评分/输入框）、锁定输入防误操作 |
| 后端 Agent | FastAPI + SSE + LLM（多轮工具调用）| 路由 Skill、调用工具、生成 Block 流式返回 |
| 数据层 | Markdown 文件（`profiles/xxx.md`）+ Session 内存 | 持久化画像、维护会话状态 |

## 一、入口与页面结构

### 1.1 路由

```
/profile-helper            → 默认展示 ChatPage（对话窗口 + 侧边画像面板）
/profile-helper/profile    → ProfilePage（结构化画像全页展示）
/profile-helper/scales     → ScalesPage（量表列表）
/profile-helper/scales/:id → ScaleTestPage（量表作答）
```

### 1.2 会话初始化（Session 生命周期）

用户首次打开页面时，前端按顺序执行：

```
1. 尝试从 localStorage 读取 auth_token → 调用认证接口（失败则继续匿名模式）
2. 从 localStorage 读取 sessionId（key: tashan_profile_session_id）
3. GET /api/profile-helper/session?session_id=xxx → 服务端创建或恢复 session
4. 从 localStorage 恢复历史消息（key: profile_helper_chat_{sessionId}）
5. GET /api/profile-helper/profile/{sessionId} → 恢复当前画像内容
```

**Session 对象在后端内存中包含：**
- `session_id`
- `messages`（完整对话历史，喂给 LLM）
- `profile`（当前画像 Markdown 内容）
- `profile_path`（磁盘路径）
- `user_id`（登录用户，匿名则 None）
- `updated_at`

## 二、核心对话机制

### 2.1 前端输入状态机

前端 ChatWindow 有三种输入状态，决定用户能否输入：

```
initial   → 页面无消息时，输入框只显示"建立我的分身"，只读，点击自动发送
locked    → AI 正在回复（loading=true）或 AI 最新消息包含 choice/rating Block
active    → 普通文字对话状态，用户可自由输入
```

**状态跃迁规则：**

```
[initial]
  用户点击发送 → 发送"建立我的分身" → locked（等待 AI）

[locked - AI 回复中]
  收到最终 Block → 判断：
    有 choice 或 rating Block → 继续 locked（提示：请点击上方选项）
    仅 text / text_input / copyable / actions → active

[active]
  用户手动发送 → locked（等待 AI）

[locked - 等待选择]
  用户点击 Block 中的按钮/选项 → 自动发送 → locked（等待 AI）
```

### 2.2 Block 协议（前后端通信格式）

前端每次发送消息走 `POST /api/profile-helper/chat/blocks`，服务端以 SSE 流式返回一系列 Block，每条 SSE 是一个完整 JSON：

```
data: {type: "text", content: "你好，我是你的画像助手..."}
data: {type: "choice", id: "start_method", question: "你想如何开始？", options: [...]}
data: [DONE]
```

**七种 Block 类型：**

| 类型 | 锁定输入？ | 说明 |
|---|---|---|
| `text` | 否 | 纯文字，Markdown 渲染 |
| `choice` | **是** | 单选按钮组，某选项可附带内联文字输入（text_prompt） |
| `text_input` | 否 | 开放文字输入框（可多行） |
| `rating` | **是** | 1~N 分评分按钮 |
| `chart` | 否 | 雷达图或柱状图（SVG 可视化） |
| `copyable` | 否 | 带一键复制按钮的文本框 |
| `actions` | 否 | 跳转按钮（如"去填写量表"） |

**关键设计**：`choice` 和 `rating` 锁定输入框，强制用户必须点击 Block 中的按钮，而非手动输入。点击后前端自动将选项内容以文字消息发送给后端，进入下一轮对话。

### 2.3 后端 Agent 循环

后端收到用户消息后，进入多轮工具调用循环（最多 40 次）：

```
用户消息进入
    ↓
检查快速路径？
  ├── 首次消息 → 直接返回欢迎 Block（不调 LLM）
  └── 消息匹配 AI 记忆触发词 → 直接返回提示词 copyable Block（不调 LLM）
    ↓
正常路径：调用 LLM（携带对话历史 + 系统 Prompt + 工具定义）
    ↓
LLM 返回工具调用 → 执行工具：
  ├── 后端工具（结果喂回 LLM 继续）：
  │   ├── read_skill(skill_name)   → 读取 SKILL.md 完整内容
  │   ├── read_doc(doc_name)       → 读取知识库文档
  │   ├── read_profile()           → 读取当前画像内容
  │   └── write_profile(content)   → 写入画像到磁盘 + 更新 session
  └── UI 工具（转换为 Block 发给前端，中断循环等待用户）：
      ├── ask_choice(...)          → choice Block
      ├── ask_text(...)            → text_input Block
      ├── ask_rating(...)          → rating Block
      ├── show_profile_chart(...)  → chart Block
      ├── show_copyable(...)       → copyable Block
      └── show_actions(...)        → actions Block
    ↓
LLM 调用了 UI 工具 → 中断循环，等待用户下一条消息
LLM 直接返回文字 → 包装为 text Block 输出，结束本轮
```

**防死循环保护**：连续 18 次仅读取操作（无 UI 交互）→ 强制让 LLM 给出阶段性结论输出。

## 四、路径 A：手动问答采集（collect-basic-info）

### 完整步骤

```
Skill 触发："帮我建立分身" / "新建档案" / "开始"
    ↓
【小闭环1：基础身份采集（5问）】
  Q1: research_stage   → choice  → 博士生/博后/助理教授/副教授/教授/其他
  Q2: primary_field    → choice+text  → 理工/生命/人文/社科+具体方向填写
  Q3: method_paradigm  → choice  → 定量实证/定性研究/理论建模/计算仿真/混合
  Q4: secondary_field  → text_input → 二级方向（可选）
  Q5: institution      → text_input → 机构（可选，涉及隐私可跳过）
每问一次 AI 等待用户 → choice 锁定 → 用户点击 → 自动发送 → 下一问
    ↓
【小闭环2：科研流程能力采集（6维度×rating）】
  每维度用 rating(1-5) 问卷：
  - 问题定义能力
  - 文献整合能力
  - 方案设计能力
  - 实验执行能力
  - 论文写作能力
  - 项目管理能力
  可分批展示（每批 2-3 维度），用户逐项评分
    ↓
【小闭环3：技术能力采集】
  ask_text → 用户填写技术栈（如：Python/PyTorch/专业领域软件）
    ↓
【小闭环4：当前需求采集】
  Q1: 主要时间占用（text_input 多行）→ 事项+描述+感受
  Q2: 核心难点与卡点（text_input 多行）
  Q3: 近期最想改变的一件事（text_input）
    ↓
后端调用 write_profile() → 写入画像 一、二、三章节
    ↓
【决策节点：下一步怎么做？】
  choice Block：
  ├── A. 填写量表（进入量表路径）
  ├── B. 让AI推断（进入推断路径）
  └── C. 先到这里，我去看一下画像
```

**每一问的小闭环结构：**
```
LLM 调用 ask_choice / ask_text / ask_rating
  → 转为 Block → 前端渲染 → 用户操作
  → 前端发送用户输入 → LLM 记录 → 调用下一个 ask_xxx
  └── 重复直到当前 Skill 全部问完
```

## 五、路径 B：AI 记忆导入（import-ai-memory）

### 分两个子步骤

#### 子步骤 B1：生成提示词（generate-ai-memory-prompt）

```
用户说"帮我生成提示词"
    ↓
LLM 调用 read_profile() → 了解已有数据
    ↓
LLM 调用 read_skill("generate-ai-memory-prompt")
    ↓
按画像完整度决定：
  ├── 空画像 → 生成「全量采集」提示词（5个模块A-E）
  └── 有部分数据 → 生成「补充」提示词（仅针对空白字段）
    ↓
show_copyable(content=提示词) → 用户一键复制
    ↓
用户去 ChatGPT/Claude 粘贴，得到 AI 回复
    ↓
用户把 AI 回复粘贴回来
```

#### 子步骤 B2：解析与整合（import-ai-memory）

```
用户粘贴 AI 回复
    ↓
快速路径检测：消息匹配 AI 记忆触发词？
  └── 是 → 直接进入 import-ai-memory Skill
    ↓
LLM 读取 SKILL.md，执行解析流程：

【阶段一：分类处理】
  ✅ 高置信度条目 → 整体确认一次（ask_choice "这些信息确认导入？"）
  ⚠️ 模糊条目    → 逐条确认（ask_choice × N）
  冲突条目        → 逐条确认，呈现冲突双方
  C模块条目       → 用户确认后写入
  ❌ 记忆不足     → 不丢弃！加入追问队列

【阶段二：写入画像】
  write_profile(content) → 分批写入已确认条目
  源注释规则：
  - ✅整体确认 → "（来源：AI记忆，整体确认）"
  - ⚠️逐条确认 → "（来源：AI记忆，已用户确认）"
  - 用户填写   → "（来源：用户补充）"

【阶段三：字段完整度扫描（F1-F7 必填门槛）】
  F1: 研究阶段
  F2: 一级领域
  F3: 方法范式
  F4: 科研流程能力（至少3维度有评分）
  F5: 技术能力（至少1行）
  F6: 主要时间占用（3.1，至少1条）
  F7: 核心难点（3.2，至少1条）
    ↓
  检查结果：
  ├── 全部填写 → 输出完整度报告，进入下一步（量表/推断）
  └── 有缺口 → 合并 ❌ 追问队列 + 空缺字段 → 进入追问流程

【阶段四：追问补充（每次 2-3 问）】
  ask_text × N → 用户逐批补充缺失字段
  每批完成后写入画像，再次扫描
  直到 F1-F7 全满足 → 才允许进入量表阶段
```

**路径 B 的核心设计约束**：
- ❌（记忆不足）条目不直接删除，进入追问队列
- F1-F7 全部满足是硬性门槛，不允许跳过
- 量表阶段在基础信息完整之前不触发

## 六、量表采集路径（三套量表小闭环）

量表有两个入口：
1. 对话中 Agent 问"下一步？"时选择"填写量表"→ 在对话窗口内逐题完成
2. 直接访问 `/profile-helper/scales/:id` → 独立量表页面作答

### 6.1 RCSS（科研认知风格量表，8题）

```
Skill: administer-rcss
    ↓
展示说明文字（text Block）
    ↓
【小闭环：8 道 rating(1-7) 题】
  A1-A4：横向整合倾向（4题）
  B1-B4：垂直深度倾向（4题）
  每题一个 rating Block → 用户点击评分 → 自动下一题
    ↓
计算：
  I = A1+A2+A3+A4（横向整合总分，4-28）
  D = B1+B2+B3+B4（垂直深度总分，4-28）
  CSI = I - D（-24 ~ +24）
  类型判定：
    CSI ≥ 10  → 强整合型
    4-9       → 倾向整合
    -3 ~ 3    → 平衡型
    -4 ~ -9   → 倾向深度
    ≤ -10     → 强深度型
    ↓
chart Block（雷达或柱状）展示结果
    ↓
write_profile() → 写入画像第四章
    ↓
更新 session: rcss_done = true
```

### 6.2 AMS（学术动机量表，28题）

```
Skill: administer-ams
    ↓
【小闭环：28 道 rating(1-7) 题，分4批展示（每批7题）】
  维度：
  - 求知内在动机（IM-KN）：题 1,8,15,22
  - 成就内在动机（IM-AC）：题 2,9,16,23
  - 体验刺激内在动机（IM-ST）：题 3,10,17,24
  - 认同调节（ID）：题 4,11,18,25
  - 内摄调节（IJ）：题 5,12,19,26
  - 外部调节（EX）：题 6,13,20,27
  - 无动机（AM）：题 7,14,21,28
    ↓
计算各维度均分（7题平均）
RAI = 3×(IM-KN+IM-AC+IM-ST)/3 + 2×ID - IJ - 2×EX - 3×AM
    ↓
chart Block（7维度柱状图 + RAI 指数）
    ↓
write_profile() → 写入画像第五章
    ↓
更新 session: ams_done = true
```

### 6.3 Mini-IPIP（大五人格，20题）

```
Skill: administer-mini-ipip
    ↓
【小闭环：20 道 rating(1-5) 题，分2批（每批10题）】
  维度（各4题，含反向题）：
  - 外向性（E）：题 1,6R,11,16R
  - 宜人性（A）：题 2R,7,12R,17
  - 尽责性（C）：题 3,8R,13,18R
  - 神经质（N）：题 4R,9,14R,19
  - 开放性（I）：题 5,10R,15,20R
  反向题处理：val = 5+1 - raw
    ↓
计算各维度均分（4题平均）
    ↓
chart Block（大五雷达图）
    ↓
write_profile() → 写入画像第六章
    ↓
更新 session: mini_ipip_done = true
```

### 6.4 独立量表页面（ScaleTestPage）流程

前端独立作答，不经 Agent：

```
进入 /profile-helper/scales/:scaleId
    ↓
展示量表信息页（不计入进度）
    ↓
开始作答 → 题目逐题展示：
  点击评分 → setAnswers[idx] → 180ms 延迟自动跳下一题
  可手动←→导航
    ↓
最后一题作答后 180ms → completed=true → 展示结果页
    ↓
用户点击「保存结果」：
  calculateScores() → POST /api/profile-helper/scales/submit
  localStorage 缓存结果
    ↓
结果同步写入画像（后端 submit 接口调用 write_profile）
```

## 七、AI 推断路径（infer-profile-dimensions）

```
Skill: infer-profile-dimensions
触发：用户说"帮我推断" / "快速估算" / 不想填量表
    ↓
LLM 调用 read_profile() → 读取基础身份 + 能力信息
    ↓
LLM 基于已有数据推断（不调额外工具）：
  - 认知风格（CSI）：基于研究方法范式推断

  - Mini-IPIP 大五：基于科研能力自评推断
    ↓
所有推断结果标注：（AI推断，置信度较低，建议后续量表验证）
    ↓
write_profile() → 写入画像四~六章（带推断标注）
    ↓
更新 session: inferred_done = true
    ↓
展示结果 chart Block（雷达图）
```

## 八、审核与修改（两个 Skill 协作）

### 8.1 review-profile

```
Skill: review-profile
触发："查看画像" / "审核" / "我想确认一下"
    ↓
LLM 调用 read_profile()
    ↓
按 7 个维度逐块展示（text Block）：
  一、基础身份 → 二、能力 → 三、需求
  → 四、认知风格 → 五、学术动机 → 六、人格 → 七、综合解读
    ↓
每个维度后询问：
  choice Block：
  ├── ✅ 这部分没问题
  ├── ✏️ 需要修改某些字段
  └── 🔁 需要重新做这个量表
    ↓
用户逐维度确认后：
  write_profile() → 写入第八章「审核记录」（日期/字段/反馈/处理方式）
  更新 session: review_done = true
```

### 8.2 update-profile

```
Skill: update-profile
触发："修改" / "更新" / "补充" / "不对" / 审核时选择「修改」
    ↓
ask_choice：要修改哪一块？（A-I 菜单）
    ↓
文字字段 → ask_text 直接询问新值
量表字段 → ask_choice：
  ├── A. 修改单个题目答案
```

## 十二、小闭环 vs 大闭环 · 总结

| 闭环 | 层级 | 定义 | 终止条件 |
|---|---|---|---|
| 单问 | 微观 | ask_xxx → 用户回答 → 写入 session | 用户提交回答 |
| Skill 内采集 | 小闭环 | N 个连续问题采集一个模块（如基础身份5问） | 当前模块所有字段采集完毕，调用 write_profile |
| 量表作答 | 小闭环 | M 道题逐题评分，完成后计算 + 写入 | 最后一题作答完 + 用户确认保存 |
| 追问补充 | 小闭环 | F1-F7 缺失字段逐批追问 | F1-F7 全部满足 |
| 隐私确认 | 小闭环 | 论坛分身生成前的6项确认 | 全部6项确认完毕 |
| 审核逐块 | 小闭环 | 7个维度逐块审核 + 反馈 | 全部维度审核完毕 |
| 画像构建 | 大闭环A | 基础信息（路径A或B）+ 量表/推断 → 初版画像 | review_done |
| 论坛发布 | 大闭环B | 画像完成 → 隐私确认 → 生成论坛分身 → 发布 | 发布到专家库成功 |
| 持续迭代 | 元闭环 | 用户可随时返回修改任意字段，触发 update-profile | 用户主动结束 |

## 十三、关键设计决策

1. **Skill 按需加载**：LLM 不预先知道 Skill 内容，通过 `read_skill(name)` 即时加载对应 SKILL.md，保持 system prompt 精简，同时 Skill 可独立迭代。

2. **UI 工具中断循环**：`ask_choice/ask_text/ask_rating` 调用后立即终止 Agent 循环，等用户回复。避免一轮内 LLM 无限自问自答。

3. **快速路径绕过 LLM**：欢迎流程和 AI 提示词生成用静态模板，节省首次 Token 延迟。

4. **F1-F7 硬性门槛**：基础信息不完整时，即使量表数据有了，也不允许进入量表阶段。防止画像出现"有人格数据但不知道这人是哪个领域"的结构性缺口。

5. **❌ 不丢弃**：AI 记忆中置信度不足的条目（❌）加入追问队列而非删除，防止信息损失。

6. **双写机制**：`write_profile` 同时更新内存 session 和磁盘文件，下次打开页面从 GET /profile 接口恢复，前端也缓存在 localStorage，三重保障不丢数据。

7. **my_twin 联动**：论坛画像写入后自动同步到 `agents/my_twin/` 目录，供圆桌讨论中的 Agent 使用，实现画像系统与圆桌讨论大闭环的连接。
