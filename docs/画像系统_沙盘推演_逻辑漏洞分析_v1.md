# 画像构建系统 · 沙盘推演 · 逻辑漏洞与技术问题分析

> 版本：v1.0 | 日期：2026-03-17
> 基于代码严格推演，不做假设，所有结论有代码出处。

## 推演前提：用户的期望行为

| 期望 | 编号 |
|---|---|
| AI在有限轮问答后采集到所有画像维度，不缺失 | E1 |
| 用户随时离开，历史对话和画像均被保存 | E2 |
| 再次登录时从上次中断处继续，而非重头来 | E3 |
| 画像完成后可归入历史，历史中可查对话过程+产出画像 | E4 |
| AI一次只问一个问题，且选择题优先 | E5 |

## 第一类漏洞：会话续接机制（E2 / E3 严重失效）

### 漏洞 L1：Session 消息历史只存内存，1小时后永久丢失

**代码位置**：`sessions.py`

```python
# 全局内存字典，进程消亡即清零
_sessions: dict[str, dict] = {}

SESSION_TTL_SECONDS = int(os.getenv("PROFILE_HELPER_SESSION_TTL_SECONDS", "3600"))

def _cleanup():
    expired = [sid for sid, s in _sessions.items() if _is_expired(s, now)]
    for sid in expired:
        _sessions.pop(sid, None)   # ← 消息历史永久删除，无磁盘备份
```

**Session 对象包含**：
- `messages`：完整 LLM 对话历史（给 AI 看的上下文）
- `profile`：当前画像 Markdown（会写磁盘）
- `scales`：量表数据（会写磁盘）

**磁盘写入情况**：
- `profile.md` ✅ 写磁盘（每次 `write_profile` 调用）
- `scales.json` ✅ 写磁盘
- `messages` ❌ **从不写磁盘**

**推演场景**：
```
用户 Q1 Q2 Q3 Q4（未触发 write_profile）→ 离开 → 1小时后回来
→ _cleanup() 删除 session
→ get_or_create(old_session_id) → 创建新 session，messages = []
→ 登录用户：从磁盘读取 profile.md（但此时 profile 还是模板，因为未触发 write_profile）
→ 匿名用户：profile 也是空模板
→ AI 完全不知道之前的 Q1-Q4 答案
→ 对话从头开始 ← 与 E3 要求完全相反
```

**严重程度**：🔴 高危，必须修复

### 漏洞 L2：前端 localStorage 消息 ≠ 后端 Session 消息，恢复后状态分裂

**代码位置**：`ChatWindow.tsx`

```typescript
// 前端从 localStorage 恢复消息（纯 UI 显示层）
const raw = localStorage.getItem(`profile_helper_chat_${sessionId}`)
if (Array.isArray(parsed) && parsed.length > 0) {
  setMessages(parsed)   // ← 只更新 UI，不同步到后端 session.messages

```

**后端 GET /session 接口**：
```python
# sessions.py: get_or_create
# 若 session_id 命中内存 → 直接返回（messages 完好）
# 若 session_id 不在内存（过期）→ _new_session()：
#   登录用户：读 profile.md + scales.json，但 messages = []
#   匿名用户：messages = []
```

**推演：1小时后用户回来**

```
前端状态：
  localStorage → messages: [Q1,A1,Q2,A2,Q3,A3]（展示完整历史）
  localStorage → sessionId: "abc123"

后端状态：
  GET /session?session_id=abc123 → 创建新 session，messages: []

用户视角：看到完整对话历史，以为可以继续
AI 视角：上下文为空，以为是全新用户

用户说"继续填写"→ AI 回复"您好！我是您的科研画像助手，您想如何开始建立分身？"
→ 用户困惑："我刚才已经填了一半了！"
```

**严重程度**：🔴 高危，造成用户体验严重断层

### 漏洞 L3：匿名用户 Session 过期后 profile.md 不恢复

**代码位置**：`sessions.py` → `_new_session()`

```python
def _new_session(session_id: str, user_id=None) -> dict:
    ...
    if user_id:
        # 登录用户 → 从 user_profile_dir 读取 profile.md ✅
        profile_path = get_user_profile_dir(user_id) / "profile.md"
        if profile_path.exists():
            profile = profile_path.read_text(encoding="utf-8")
    else:
        # 匿名用户 → profile = 模板内容（不搜索磁盘已有文件）❌
        profile = _load_template()
    ...
```

**推演**：
```
匿名用户完成了画像 → write_profile 写入 profiles/zheng-abc12345.md
→ 1小时后回来 → 新 session 创建 → profile = 模板（空）
→ 磁盘上 zheng-abc12345.md 存在，但永远不会被找到
→ 用户看到的是空画像，以为数据丢了
```

**根本原因**：匿名用户的 profile 文件名包含 session_id 前8位，理论上可通过 session_id 反推文件名，但代码没有实现此逻辑。

**严重程度**：🟠 中高危（匿名用户全部受影响）

## 第二类漏洞："一次一问"未强制执行（E5 失效）

### 漏洞 L5：LLM 单次响应可调用多个 ask_xxx，前端渲染多个交互 Block

**代码位置**：`block_agent.py`

```python
for tc in tool_calls:   # ← 遍历所有工具调用，没有限制个数
    if tc.function.name in _INTERACTIVE_UI_TOOLS:
        block = _ui_tool_to_block(...)
        response_blocks.append(block)   # ← 直接追加，不检查是否已有一个交互 Block
        has_interactive_ui = True
    ...
if has_interactive_ui:
    break   # ← 循环 break，但本次 iteration 内的所有 Block 都已添加
```

**结果**：若 LLM 调用 `ask_rating("问题定义能力")` + `ask_rating("文献整合能力")` 在同一次响应中，用户会看到两个评分 Block。

**这种情况是否真的会发生？**

`collect-basic-info/SKILL.md` 明确写道：
```
第二批：科研流程能力评分
- 每次询问 2 个维度，分 3 轮完成
- 第一轮：问题定义能力 + 文献整合能力
```

这意味着 LLM **被要求** 在一次响应中发出 2 个 rating Block。

**严重程度**：🟠 中高危（数据丢失，见 L6）

### 漏洞 L6：多个交互 Block 同一消息时，只有第一个答案被捕获

**代码位置**：`ChatWindow.tsx` → `handleBlockRespond`

```typescript
const handleBlockRespond = (messageIndex: number, responseText: string) => {
    // 1. 标记该消息为已响应
    setMessages(prev => prev.map((m, i) =>
        i === messageIndex ? { ...m, _responded: true } : m
    ))
    // 2. 发送用户选择的内容
    handleSend(responseText)
}
```

**前端 Block 禁用逻辑**：
```typescript
const isResponded = message._responded
const isLatest = i === messages.length - 1
disabled={loading || (isInteractive && (!isLatest || isResponded))}
```

**推演**：消息中有 Block[0]=rating("问题定义") + Block[1]=rating("文献整合")

```
用户点击 Block[0] 的评分 → handleBlockRespond(idx, "4") 被调用
  → 整条消息 _responded = true
  → handleSend("4") 发送 → AI 收到 "4"（问题定义=4）
  → 下一轮 AI 继续
此时 Block[1] 的状态：
  isResponded = true（整条消息已标记）→ disabled = true
  用户无法点击 Block[1]，"文献整合能力"评分永远丢失
```

**数据完整性结论**：
- 每条消息中只有 **第一个** 交互 Block 的答案可以被提交
- 第二个及之后的交互 Block 被永久禁用
- 这与 SKILL.md 要求"每次2维度评分"形成直接冲突

**严重程度**：🔴 高危，数据丢失

### 漏洞 L7："一次一问"和"选择题优先"无全局代码约束

**代码位置**：无（以下是对代码缺失的描述）

- `block_agent.py` 的系统 Prompt（`META_SYSTEM_PROMPT`）：无"一次一问"约束条款
- `agent.py` 的系统 Prompt：无"选择题优先"条款
- 各 Skill SKILL.md：仅 `collect-basic-info` 写了"逐个提问"，其他 Skill 无此约束
- `profile-collector.mdc`（Cursor rules 层）：与 Web 应用 runtime 无关

**影响范围**：
- `review-profile`：7个维度逐块审核，AI 可能一次发多块询问
- `update-profile`：AI 可能同时展示多个修改选项
- `import-ai-memory`：确认环节 AI 可能一次列多个条目

**严重程度**：🟡 中危（依赖 LLM 概率行为，非确定性失效）

## 第三类漏洞：信息采集完整性（E1 失效）

### 漏洞 L8：block_agent 最大迭代次数硬编码 20 次，复杂流程会被截断

**代码位置**：`block_agent.py`

```python
max_iterations = 20   # ← 硬编码，无法通过环境变量覆盖
```

对比：`agent.py`（旧端点）：
```python
MAX_TOOL_ITERATIONS = max(5, int(os.getenv("PROFILE_HELPER_MAX_TOOL_ITERATIONS", "40")))
```

**import-ai-memory 的典型工具调用分析**（复杂用户场景）：

```
1. read_skill("import-ai-memory")          → 1次
2. read_profile()                           → 1次
3. ask_choice("阶段一：确认✅条目")         → 1次（中断）
4. [用户回复] ask_choice("⚠️条目1")        → 1次（中断）
5. [用户回复] ask_choice("⚠️条目2")        → 1次（中断）
6. [用户回复] ask_choice("冲突条目")        → 1次（中断）
7. [用户回复] write_profile()               → 1次（第一次写入）
8. read_profile()（再次读以做F1-F7扫描）    → 1次
9. ask_text("F6缺失：时间占用")             → 1次（中断）
10. [用户回复] ask_text("F7缺失：核心难点") → 1次（中断）
11. [用户回复] write_profile()              → 1次（第二次写入）
--- 以上是 11 轮，每轮一次用户交互 ---
```

但每轮用户交互是一次新的 HTTP 请求，都有独立的 20 次迭代限制。

**真正的风险**：同一轮（用户一次发送）内，LLM 需要做多件事：
```
read_skill → read_profile → 分析内容 → read_doc（如有） → ask_choice
= 至少 4 次工具调用（1次read_skill + 1次read_profile + 可能的read_doc + 1次ask_xxx）
```

如果 AI 多次 read_doc 或 read_profile，加上处理逻辑复杂，20 次可能不够，触发截断：
```python
# 超出 max_iterations 时的处理（block_agent.py）
# ... 代码中没有明确的熔断保护（agent.py 有，block_agent.py 无）
return response_blocks   # ← 此时 response_blocks 可能只有部分结果
```

**严重程度**：🟡 中危（罕见但可触发）

---

### 漏洞 L9：F1-F7 完整度门槛是 LLM 的软约束，无后端代码强制

**代码位置**：`import-ai-memory/SKILL.md`（文本指令）

```markdown
5.4 根据完整度决定下一步
  - F1-F7 全部满足 → 才可进入量表选择
  - 有任何缺口 → 继续追问，不允许跳过
```

**实际后端实现**：
- 无任何 API 端点验证 F1-F7 是否完整
- 无任何后端函数解析画像文件并检查字段
- 上述逻辑完全依赖 LLM 在 Skill 指令下自觉执行

**失效场景**：
- 用户直接说"跳过，我就要做量表" → LLM 可能在 Skill 约束和用户意愿之间妥协，开放量表入口
- 对话轮数过多后，LLM 可能"忘记"F1-F7 门槛约束
- 用户切换到手动输入"我要填量表了"，绕过 import-ai-memory 的后续检查

**严重程度**：🟡 中危（概率性失效）

### 漏洞 L10：LLM 路由是概率性软约束，可能走错 Skill 或直接回答

**代码位置**：`block_agent.py` → `META_SYSTEM_PROMPT`

```python
# 系统 Prompt 描述的路由规则（文字形式，非代码逻辑）：
# "用户说'帮我建立分身' → 调用 read_skill('collect-basic-info')"
# "用户粘贴 AI 回复 → 调用 read_skill('import-ai-memory')"
```

**快速路径（代码层的唯一确定性路由）**：
```python
if _is_fresh_session_or_init_msg(session, user_message):  → 欢迎模板
if _is_ai_memory_request(user_message, session):           → AI记忆提示词
```

**其他所有情况**：完全依赖 LLM 概率决策

**失效形式**：
1. LLM 直接回答而不调用 `read_skill` → Skill 指令不生效 → 问题节奏/格式混乱
2. LLM 调用错误的 Skill → 流程走错分支
3. LLM 重复调用 `read_skill`（连续 18 次触发 agent.py 熔断，但 block_agent.py 无此保护）

**严重程度**：🟡 中危

## 第四类漏洞：历史记录不闭环（E4 失效）

### 漏洞 L11："历史分身"只有已发布的分身，未发布画像无法进历史

**代码位置**：`ProfilePage.tsx`

```typescript
// Section 6: 历史分身记录
const twins = await authApi.getDigitalTwins(token)
// → 调用 GET /api/auth/digital-twins（TopicLab 账号系统）
// → 返回已通过 POST /publish-to-library 发布的分身
```

**什么情况下会有历史记录**：用户点击"发布到他山圆桌" → `POST /publish-to-library`

**什么情况下不会有历史记录**：
- 用户建立了完整画像但没发布 → 无历史
- 用户建立了部分画像 → 无历史
- 用户做了第一次完整画像，想看历史，再建第二次 → 第一次的记录必须先发布才能查看

**与期望 E4 的差距**：
- E4 要求"建立完了就归入历史记录，可查对话过程+产出画像"
- 实际："必须主动发布才有历史，且历史只含画像内容，不含对话过程"

**严重程度**：🟠 中高危（功能缺失）

### 漏洞 L12：对话过程（conversation log）完全不可查看

**代码位置**：无（以下是对代码缺失的描述）

**严重程度**：🟡 中危（功能缺失）

## 修复方向（按优先级）

### P0：立即修复（数据丢失类）

**L1 + L2 根治方案**：将 messages 持久化到磁盘

```python
# sessions.py 新增
def _save_messages(session: dict):
    """每次消息更新后保存到磁盘"""
    path = _get_messages_path(session)   # profiles/.messages/{session_id}.json
    path.write_text(json.dumps(session["messages"], ensure_ascii=False), encoding="utf-8")

def _load_messages(session_id: str) -> list:
    """创建 session 时尝试从磁盘恢复 messages"""
    path = _get_messages_path_by_id(session_id)
    if path.exists():
        return json.loads(path.read_text(encoding="utf-8"))
    return []
```

Session 过期后恢复顺序：`profile.md → messages.json → scales.json`，三者全部恢复，AI 可以真正从断点续接。

**L6 根治方案**：`_responded` 改为 Block 级而非消息级

```typescript
// 方案：messages 中每条消息记录 responded_block_ids: Set<string>
// 只有该 block.id 在 responded_block_ids 中才 disabled
```

## 总结

当前画像系统在 **E1（信息完整性）** 和 **E3（断点续接）** 两个维度存在根本性的架构缺陷，核心原因是：

> **会话状态（messages）只存内存，没有任何持久化机制，1小时 TTL 后 AI 上下文彻底丢失。**

前端用 localStorage 展示历史对话，给用户造成"数据保存了"的假象，但后端 AI 已经失忆，无法真正续接。这是最高优先级的修复项。

其次，**多交互 Block 并存 + _responded 消息级标记** 造成评分数据丢失，也需立即修复。

其他漏洞（F1-F7 软约束、历史记录缺失、匿名画像丢失）属于功能不完整，可按优先级排期修复。
