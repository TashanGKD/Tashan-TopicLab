## 一、用户可见的问题修复说明

> 以下用非技术语言描述每个问题和修复效果，适合产品、运营和用户理解。

---

### 🔴 问题1：离开超过1小时后，AI会"失忆"，重头开始问

**问题描述**

用户和 AI 对话了一半（比如已经回答了研究阶段、学科领域），关闭浏览器或网络断开。1小时后再回来，界面上能看到之前的对话记录，但 AI 对之前说过的一切毫无印象，重新从"你好，您想如何建立分身？"开始。

**根本原因**

对话历史（AI的记忆）只存在服务器内存里，服务器1小时会自动清理。界面上显示的历史是浏览器本地存储的，两者不同步。

**修复效果**

现在每轮对话结束后，AI 的完整对话历史会立即写入磁盘文件。用户无论何时回来，AI 都能从磁盘恢复记忆，真正从上次中断的地方继续，而非重头开始。

---

### 🔴 问题2：评分题一次出现两个，只有第一个答案被记录

**问题描述**

在"科研流程能力"评分环节，AI 有时会同时展示两道评分题（如"问题定义能力"和"文献整合能力"各一个1-5分的题目）。用户点击第一道题的评分后，第二道题立刻变灰、无法点击。第二道题的评分从未被记录，导致能力评估数据缺失。

**根本原因**

AI 可以在同一轮响应中同时发出多个问题组件，前端在用户点击第一个之后，就把整条消息标记为"已响应"，第二个组件随即被禁用。

**修复效果**

两处联动修复：
1. 后端限制：AI 现在每轮响应最多只能出一个问题，多余的问题被系统自动拦截，下一轮再问；
2. 前端升级：响应状态从"消息级"改为"问题级"，每道题独立追踪，互不干扰。

---

### 🟠 问题3：没有账号的用户，超过1小时后画像数据丢失

**问题描述**

匿名（未登录）用户花了30分钟建立画像，写入了研究阶段、能力评分等数据。但超过1小时后再打开系统，看到的是一个空白的画像模板，之前填写的所有内容消失了。

**根本原因**

系统服务器到期后会重新初始化匿名用户的 session，不知道去哪里找之前保存的画像文件。

**修复效果**

现在服务器会按照用户的 session 标识主动查找已有的画像文件，匿名用户超时后回来，已保存的画像数据会被正确恢复。

---

### 🟠 问题4：回答几个问题后意外断线，所有答案消失

**问题描述**

用户在"手动问答"模式下，回答了Q1研究阶段、Q2学科领域、Q3研究方法，在回答Q4机构时网络断开。重新进来后，AI记录的画像文件是空的，Q1-Q3的答案全部丢失。

**根本原因**

AI 采集到所有问题全部答完，才会统一将数据写入画像文件。中途断线，已经回答的内容全部随 session 消失，一个字都没保存。

**修复效果**

画像采集改为"分批写盘"模式：每完成一批问题（如基础身份Q1-Q5全部完成），立即写入画像文件。回来后 AI 读取已有画像，知道哪些批次已经完成，从断点继续采集剩余内容。

---

### 🟠 问题5：AI有时一次发出多个问题，体验混乱

**问题描述**

AI 有时会一次性发出两个问题（如一个选择题加一个文字输入框），用户不清楚先回答哪个，交互体验混乱。

**根本原因**

"每次只问一个问题"的规则只写在 AI 的行为指南里，没有代码层面的强制保障。AI 偶尔会忽略这条规则。

**修复效果**

双重保障：代码层面强制拦截每轮第二个及之后的问题组件；同时在 AI 的系统提示中明确告知"系统会在代码层拦截多余问题"，让 AI 理解这是硬性限制，不再尝试同时发多个问题。

---

### 🟡 问题6：复杂流程（如AI记忆导入）可能被中途截断

**问题描述**

在"导入AI记忆"模式下，系统处理步骤较多（解析、逐条确认、写入、完整度检查），偶发出现流程被截断、AI 没有正常完成整个导入过程的情况。

**根本原因**

后端工具调用次数上限被硬编码为20次，复杂流程有超出限制的风险。

**修复效果**

上限改为默认30次，并可通过服务器环境变量 `PROFILE_HELPER_BLOCK_MAX_ITERATIONS` 灵活调整。

---

### 🟡 问题7：匿名用户登录后，之前建立的画像数据丢失

**问题描述**

用户在没有登录的状态下建立了半个小时的画像，然后决定注册并登录账号。登录后，系统创建了新账号对应的空白画像，之前匿名时建立的画像数据再也找不回来。

**根本原因**

用户登录后，系统为新账号创建全新的画像存储路径，不会自动寻找和关联之前的匿名画像文件。

**修复效果**

现在用户登录时，系统会自动检测：如果该账号还没有画像，且当前 session 中有匿名建立的画像内容，则自动将匿名画像迁移到该账号下，不丢失任何数据。

---

### 🟡 新功能：支持查看完整对话历史

**功能说明**

新增 API 接口，可以读取用户与 AI 的完整对话记录（过滤掉后台系统消息，只保留用户和 AI 之间有意义的对话内容）。为后续"历史对话查看"功能提供数据支撑。

---

### 🟡 新功能：画像完整度自动检测工具

**功能说明**

AI 现在拥有一个名为 `check_profile_completeness` 的工具，可以主动扫描当前画像文件，精确判断7个必填字段（研究阶段、一级领域、方法范式、科研流程能力、技术能力、时间占用、核心难点）是否全部填写完整，并列出缺失字段清单。这确保了 AI 不会在画像不完整时就进入量表阶段。

### 修改文件六：`frontend/src/modules/profile-helper/ChatWindow.tsx`

**变更类型**：修改函数签名 · 修改状态更新逻辑 · 修改渲染逻辑

#### 1. `handleBlockRespond` — 函数签名新增 `blockId` 参数

```typescript
// 修改前
async (msgIndex: number, responseText: string)

// 修改后
async (msgIndex: number, responseText: string, blockId?: string)
```

#### 2. `handleBlockRespond` — 状态标记改为 Block 级

**修改前**（消息级，点击任意 block 后整条消息被锁）：
```typescript
next[msgIndex] = { ...msg, _responded: true }
```

**修改后**（Block 级，每个 block 独立标记）：
```typescript
const respondedBlocks = [...(msg._responded_blocks ?? [])]
if (blockId && !respondedBlocks.includes(blockId)) {
  respondedBlocks.push(blockId)
}
const interactiveIds = (msg.blocks ?? [])
  .filter((b) => b.type === 'choice' || b.type === 'rating' || b.type === 'text_input')
  .map((b) => ('id' in b ? (b as { id: string }).id : ''))
  .filter(Boolean)
const allResponded = interactiveIds.length > 0 && interactiveIds.every((id) => respondedBlocks.includes(id))
next[msgIndex] = { ...msg, _responded_blocks: respondedBlocks, _responded: allResponded }
```

#### 3. 渲染侧 `BlockRenderer` — `disabled` 和 `responded` 改为 Block 级

**修改前**（消息级，所有 block 共享同一个 `isResponded`）：
```typescript
const isResponded = !!m._responded
// ...
disabled={loading || (isInteractive && (!isLatest || isResponded))}
responded={isResponded}
```

**修改后**（Block 级，每个 block 独立判断）：
```typescript
const respondedBlocks = m._responded_blocks ?? []
// 对每个 block 单独计算：
const blockId = 'id' in block ? (block as { id: string }).id : undefined
const isBlockResponded = blockId ? respondedBlocks.includes(blockId) : !!m._responded
// ...
onRespond={(text) => handleBlockRespond(i, text, blockId)}
disabled={loading || (isInteractive && (!isLatest || isBlockResponded))}
responded={isBlockResponded}
```
