## 一、协作原则（关键，先理解再执行）

### 核心流水线（不可跳步）

```
产品经理产出PRD
    ↓
关卡A：用户模拟审核（role-审核者-用户模拟）
    ↓ 通过
技术架构师产出架构方案
    ↓
确认：PM 先看架构，确认不偏离产品意图
    ↓
关卡B：系统破坏审核（role-审核者-系统破坏）
    ↓ 通过
UI设计师产出设计稿 ←→ 前端/后端/AI工程师 并行开发
    ↓
关卡C：测试工程师验收（role-测试工程师）
    ↓ 通过
DevOps 部署上线
    ↓
数据分析师监控断裂节点 → 反馈回 PM
```

### 本次特殊说明

本次已有：
- ✅ 产品设计文档（相当于 PM 产出了 PRD 的初版）
- ✅ 工程实施计划（相当于技术架构师产出了实施方案的初版）

**因此，当前最合理的起点是：先过关卡A和关卡B，确认没有设计漏洞，再启动开发。**

### 阶段 1：关卡A——用户模拟审核（⚠️ 写代码前必须过）

**角色**：`role-审核者-用户模拟`  
**触发时机**：产品设计文档完成后，代码开始之前  
**目的**：找出设计中的用户动线断点

#### 提示词（直接使用）

```
请读取以下两个文档，然后以 role-审核者-用户模拟 的视角执行关卡A审核：

文档1：@docs/产品设计_数字分身与圆桌讨论_v1.md
文档2：@docs/工程实施计划_数字分身与圆桌大闭环_v1.md

执行关卡A的完整流程：
1. 提取「画像→圆桌」这条新增大闭环的核心用户路径
2. 分别扮演以下4类用户走一遍路径：
   - 第一次用的用户（完全不知道逻辑，靠猜测行动）
   - 懒惰用户（想用最少点击完成任务）
   - 挑剔用户（对每个设计选择挑毛病）
   - 走偏路的用户（不按设计意图使用）

3. 逐条执行通用必问清单 A-01 到 A-10
4. 逐条执行他山特有问题 AT-01 到 AT-04，特别重点审核：
   - AT-01：用户完成分身后，下一步引导是否足够清晰？
   - 新增：「DefaultExpertPanel」按钮点击后到「进入圆桌」的中间步骤是否对用户直觉？
   - 新增：「判断行为捕获」（标记按钮）是否会干扰正常阅读讨论的体验？

5. 验证闭环六条件：
   画像建立 → 默认专家团 → 圆桌配置预填 → 讨论 → 摘要贡献可见 → 画像更新

6. 输出关卡A审核报告（按模板格式）

注意：你不是产品经理，不要为方案辩护，
要主动挑战方案在各种压力下是否仍然成立。
```

**预期输出**：关卡A审核报告，列出设计漏洞（如有）  
**后续处理**：有漏洞 → PM修复产品设计文档 → 重新过关卡A；无漏洞 → 进入阶段2

### 阶段 2：关卡B——系统破坏审核（⚠️ 写代码前必须过）

**角色**：`role-审核者-系统破坏`  
**触发时机**：工程实施计划（技术架构）完成后，PM确认后，代码开始之前  
**目的**：找出架构中的静默失败点和安全漏洞

#### 提示词（直接使用）

```
请读取以下文档，然后以 role-审核者-系统破坏 的视角执行关卡B审核：

文档：@docs/工程实施计划_数字分身与圆桌大闭环_v1.md
现有代码参考：
  - @topiclab-backend/app/api/profile_helper.py（现有端点参考）
  - @backend/app/services/profile_helper/scientist_match.py（现有算法参考）

重点审核以下新增设计的架构风险：

【新增API设计风险】
B-new-1：GET /profile-helper/profile/{session_id}/recommended-panel
  - 这个接口依赖 famous scientist 匹配和 field recommendations，
    如果这两个子服务任何一个失败，接口会崩溃还是优雅降级？
  - 如果 profile 数据不完整（CSI=0，没有认知风格数据），
    推荐结果会是什么？会报错还是返回合理的默认值？

B-new-2：POST /topics/{topic_id}/discussion/judgment
  - 这个接口在 topiclab-backend，但画像更新在 backend（Resonnet）
    跨服务调用的失败场景：Resonnet 服务不可用时，判断记录是否会丢失？
  - background_tasks 的异步更新：如果画像更新失败，用户有感知吗？

B-new-3：localStorage 数据传递（profilePanel）
  - 用户从画像页跳转到创建话题，中间的 localStorage 数据
    如果用户长时间挂起（localStorage 过期或清除），会发生什么？
  - 多个标签页同时打开，会不会相互污染？

【现有架构风险回顾】
请执行 B-01 到 B-12 的标准检查清单，重点是：
  - B-05：新的 judgment 记录是否有 user_id 隔离？
  - B-06：新增 SSE 端点（如果有）是否有全局 try/except？
  - B-11：新的 judgment_history 字段加入画像后，
          6个月后如何演化（结构变更兼容性）？

执行三类破坏者视角：
1. 黑客：能否通过 topic_id 访问别人的判断记录？
2. 极端用量：用户发起100次讨论，judgment_history 数组无限增长怎么办？
3. 六个月维护者：新人接手 judgment_updater.py，能理解 _infer_questioning_style 的逻辑吗？

输出关卡B审核报告（按模板格式）。
```

**预期输出**：关卡B审核报告，列出致命漏洞（如有）  
**后续处理**：有致命漏洞 → 架构师修复工程实施计划 → 重新过关卡B；无致命漏洞 → 进入阶段3

### 阶段 4b：后端开发——P1 判断行为记录 API

**角色**：`role-后端开发`  
**触发时机**：P0 验收通过后  
**目的**：实现判断行为的存储和画像更新逻辑

#### 提示词（直接使用）

```

读取参考文档：
  @docs/工程实施计划_数字分身与圆桌大闭环_v1.md（第三章 P1-3）
  @topiclab-backend/main.py（了解 router 注册方式）
  @topiclab-backend/app/api/topics.py（参考认证和数据存储方式）

【任务1】新建 topiclab-backend/app/api/judgment.py

实现 POST /topics/{topic_id}/discussion/judgment 端点：
- 接受 payload：
  {
    "session_id": "...",（可选，有则触发画像更新）
    "marked_turns": [...],
    "followup_questions": [...],
    "final_tendency": "..."（可选）
  }
- 存储到 DB（新增 discussion_judgments 表，字段见下）
- 如果有 session_id 且用户已登录：
  通过 Resonnet client 异步触发画像更新（不等待结果）
- 返回 {"ok": true}

discussion_judgments 表结构：
  id SERIAL PRIMARY KEY,
  topic_id VARCHAR NOT NULL,
  user_id INTEGER,
  session_id VARCHAR,
  marked_turns JSONB DEFAULT '[]',
  followup_questions JSONB DEFAULT '[]',
  final_tendency TEXT,
  created_at TIMESTAMPTZ DEFAULT now()

注意：
- 用户未登录时，仍然存储记录（user_id=null）
- 跨服务调用 Resonnet 失败时，不影响当前请求返回

【任务2】在 topiclab-backend/main.py 注册新 router

在现有 router 注册后追加：
from app.api import judgment as judgment_router
app.include_router(judgment_router.router, tags=["judgment"])

【任务3】新建 backend/app/services/profile_helper/judgment_updater.py

实现 update_profile_from_judgment(session_id, topic_id, judgment_data)：

核心逻辑：
1. read_profile(session_id) 读取画像
2. 从 judgment_data 推断认知偏好（调用 _infer_questioning_style）
3. 构建 judgment_record，追加到画像的 judgment_history 列表
4. 每积累5条记录，重新生成 cognitive_evolution 摘要
5. write_profile(session_id, 更新后的画像)

_infer_questioning_style 规则（关键词匹配）：
- 追问内容包含「方法/统计/实验设计/对照」→ 返回"方法论层面"
- 追问内容包含「理论/假设/前提/模型」→ 返回"理论基础层面"  
- 追问内容包含「数据/样本/结果/测量」→ 返回"数据与实证层面"
- 其他 → 返回"综合性追问"

实现规范：
- 所有 DB 写入函数写完后立即测试
- judgment_history 有上限（最多保存50条，超出后删除最早的）
- 完成后提交测试工程师
```

```
  E. 在专家Tab点击「应用专家团」的用户数
  F. 成功启动圆桌讨论的用户数

如果 C/B < 20%：DefaultExpertPanel 组件吸引力不足（设计问题）
```

## 四、关键规则备忘

**任何时候都不能跳过的原则：**

1. **关卡A（用户模拟）必须在写代码前通过** → 没过不开始写代码
2. **关卡B（系统破坏）必须在写代码前通过** → 没过不开始写代码
3. **技术架构方案必须先回PM确认** → 架构师完成后，PM确认符合产品意图，才能下发开发
4. **测试发现的「产品偏差」回PM** → PM决策改实现还是改PRD，开发不自行决定
5. **后端每写完DB函数，立即集成测试** → 写完≠完成，必须有实际调用证明

**本次特别注意的风险（来自关卡B预审）：**

- `recommended-panel` 接口必须有完整的降级处理（子服务失败不能返回500）
- `localStorage` 传递 profilePanel 必须考虑清除场景（跨页面/跨标签页安全）
- `judgment_updater` 的异步调用失败必须对用户静默处理，不影响讨论本身
