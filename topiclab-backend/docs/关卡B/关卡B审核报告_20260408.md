## 致命漏洞（必须修复才能开始写代码）

| # | 漏洞描述 | 检查项 | 修复建议 | 严重程度 |
|---|---|---|---|---|
| 1 | **GitHub 回调无 state 验证实现细节缺失**：方案提到生成 state 存入「内存缓存」，但未指定缓存实现。多实例部署（多进程/多容器）时，实例A生成的 state 无法被实例B验证，导致所有 GitHub 登录失败或完全绕过 CSRF 防护 | B-02 | state 必须存入 Redis 或数据库（建议 verification_codes 表新增 type='oauth_state'），不能用进程内存 |
| 2 | **`/auth/github/callback` 端点无认证保护，可被任意人枚举调用**：攻击者伪造 code 参数反复调用，可触发大量 GitHub API 请求，导致账号的 GitHub OAuth App 速率限制被耗尽。方案未描述失败次数限制 | B-02 | 增加 state 验证（修复1）+ 对同一 IP 的回调请求加频率限制（5次/分钟）|
| 3 | **`users.phone` 改为 nullable 后，`create_jwt_token(user_id, phone=None)` 的处理方式未明确**：现有代码 `create_jwt_token(user_id: int, phone: str, ...)` 中 phone 是 `str`，强类型约束。若代码层不更新，传入 None 会导致 JWT 生成失败或类型错误，且 `verify_jwt_token` 解析时也可能崩溃 | B-03 | 明确 phone 参数改为 `Optional[str]`，JWT payload 中 phone 字段允许为 null，所有调用方（包括 `/auth/me`）需适配 `phone: str \| None` |

## 已记录风险（有规避方案，可接受）

| # | 风险描述 | 严重程度 | 规避方案 |
|---|---|---|---|
| R-01 | **密码重置无防暴力破解**：`POST /auth/reset-password` 未限制错误验证码尝试次数，攻击者可暴力枚举6位数字验证码（最多100万次，验证码5分钟内有效）| 高 | 加「同一手机号验证码验证失败≥5次即锁定」逻辑；或将验证码位数增加到8位 |
| R-02 | **验证码已使用后未删除**：方案描述「可选」删除已使用验证码，但若不删除，同一验证码在5分钟内可被重复使用（重置密码后再次重置）| 中 | 必须删除（或标记为已使用）——不应是可选行为 |
| R-03 | **GitHub access_token 明文存储**：方案说「首期明文存储」，但 access_token 具有账号读取权限，一旦数据库泄露即可访问所有 GitHub 用户的基本信息 | 中 | 首期可接受，但必须在 `oauth_accounts` 表注释中标注「待加密，使用 OAUTH_TOKEN_KEY 环境变量」，并在开发完成后立即跟进 |
| R-04 | **`users.phone` 加 UNIQUE 约束 + nullable 冲突**：PostgreSQL 中 UNIQUE 约束对 NULL 值的处理是「多个 NULL 互不冲突」（每个 NULL 被视为不同值），但 SQLite 行为一致。MySQL 行为相反。文档未说明此边界 | 低 | 记录：当前使用 PostgreSQL，多个 phone=NULL 的 GitHub 用户共存没有问题（每个 NULL 不冲突）。如未来迁移数据库需重新验证 |
| R-05 | **GitHub 用户无手机号，忘记密码功能对其无效**：方案未说明 GitHub 用户如果想「重置密码」应该怎么做 | 低 | 首期可接受。在 `/forgot-password` 页面加说明「GitHub 登录用户无需密码」；后续如需支持 GitHub 用户绑定手机号，单独立项 |
| R-06 | **前端 `/auth/callback` 路由接收 token via URL 参数**：token 会出现在浏览器历史记录、服务器日志、Referer 头中。属于 OAuth implicit flow 的已知安全风险 | 中 | 首期可接受（GitHub 官方示例也使用此模式）。更安全方案是后端 Set-Cookie httpOnly，但改动更大。记录为技术债 |
| R-07 | **6个月后可演化性**：`oauth_accounts` 表的 `access_token` 列存储 GitHub token，未来若支持 Google/微信，不同 provider 的 token 格式、刷新逻辑差异极大，同一列无法通用 | 低 | `oauth_accounts` 表已按 provider 分行存储，每个 provider 独立一行，扩展时只需新增 provider='google' 等记录，不影响现有数据 |

## 通过的检查项

- **[B-01] ✅ JSONB 参数**：新增的 SQL 均使用参数化查询（`:phone`, `:code` 等），无字符串拼接，无 JSONB 相关操作
- **[B-05] ✅ 数据隔离**：`oauth_accounts` 通过 `user_id` 外键关联 `users`，查询有 `WHERE provider=? AND provider_user_id=?` 过滤，不存在跨用户数据泄露
- **[B-09] ✅ Schema 引用**：无跨 schema 引用，所有表在同一 database 下
- **[B-10] ✅ 架构-实现对齐**：改动范围清晰（两个文件、两个端点），不依赖未实现的模块
- **[B-11] ✅ 可演化性（oauth_accounts）**：`provider` 字段设计允许未来扩展 Google/微信，无需修改现有代码
- **[B-12] ✅ 环境变量验证**：`GITHUB_CLIENT_ID` / `GITHUB_CLIENT_SECRET` 需在启动时检测缺失并立即报错（方案中已要求，代码实现时需落实）
- **[B-02（部分）] ✅ 忘记密码认证边界**：`reset-password` 端点无需登录 token（正确，用验证码替代），`send-code` 已有频率限制（1分钟内不重复发送）

## 黑客视角额外检查

**攻击1：重置任意用户密码**

```
攻击链：POST /auth/send-code {phone: 受害者手机, type: reset_password}
        → 短信发到受害者手机（攻击者无法获取）
        → 攻击者无法继续
结论：✅ 安全，验证码在受害者手机上
```

**攻击2：GitHub 账号劫持**

```
攻击链：构造 ?code=valid_code&state=invalid_state → 后端验证 state 失败 → 拒绝
        但若 state 存内存（多实例）→ 验证可能失效 → 致命漏洞1
结论：❌ 需修复（致命漏洞1）
```

**攻击3：绕过手机验证码注册**

```
攻击链：已有 register_sms_bypass_active() 开关，本次改动未引入新绕过点
结论：✅ 无新绕过
```

## 结论

**不通过 ❌**：存在 3 个致命漏洞，需修复后才能开始写代码：

1. **GitHub OAuth state 必须持久化存储**（数据库或 Redis），不能用内存缓存
2. **`/auth/github/callback` 需加频率限制**
3. **`create_jwt_token` 和相关代码需明确处理 `phone=None`**

修复以上 3 点后，5 个「已记录风险」中：
- R-01（暴力破解）和 R-02（验证码重用）建议在同一批次修复（改动成本低）
- R-03、R-04、R-05、R-06、R-07 首期可接受，记录为技术债
